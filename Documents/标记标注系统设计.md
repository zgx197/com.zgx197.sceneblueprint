## 一、问题背景与设计动机

### 1.1 现状

SceneMarker 系统中，Marker 是纯空间标记：

- `PointMarker`：position + rotation
- `AreaMarker`：polygon / box 区域
- `EntityMarker`：Prefab 实例位置

Blueprint 节点通过 `SceneBinding`（MarkerId）引用 Marker，描述运行时流程。

### 1.2 新需求

在"预设刷怪"场景中，策划需要：

1. 给每个 PointMarker 指定**怪物 ID、等级、初始行为**等标注信息
2. 同一个 PointMarker 可能同时被**刷怪节点和相机节点**引用
3. 有些 PointMarker 在 AreaMarker 下（批量生成），有些独立存在（手动放置）
4. 标注信息属于 **SceneView 层**（空间标注），不属于 Blueprint 层（运行时逻辑）

### 1.3 设计约束

来自 [SceneBlueprint核心设计原则](SceneBlueprint核心设计原则.md)：

| 约束 | 来源 |
|------|------|
| Marker 只描述"是什么"，不描述"做什么" | §1.2 职责分离 |
| Blueprint 只引用 Marker ID，不存储配置数据 | §1.3 单向数据流 |
| 同一个 Marker 可以被多个节点引用 | §2.1 标记与节点关系 |
| Marker 中不存储复杂逻辑配置 | §3.3 反模式 2 |

### 1.4 关键洞察

区分三种数据：

```
空间数据（SceneView 层）：position, rotation, 区域形状
标注数据（SceneView 层）：monsterId, level, behavior, cameraFOV
逻辑数据（Blueprint 层）：时序、条件、连线关系
```

**"这个点位放骷髅战士"是标注数据**——策划在场景中做的空间标注，和"这个区域叫 Boss 房间"是同一层次。它不是运行时逻辑。

## 二、核心设计原则

### 2.1 组件化标注（路径 C）

采用 Unity 组件化方式：通过额外的 MonoBehaviour 组件附加标注数据。

```
PointMarker（空间）     +  SpawnAnnotation（标注）     = 刷怪点
PointMarker（空间）     +  CameraAnnotation（标注）    = 相机机位
PointMarker（空间）     +  SpawnAnnotation + CameraAnnotation = 刷怪点 + 相机机位
PointMarker（空间）     +  （无标注）                   = 纯空间点
```

### 2.2 设计优势

| 优势 | 说明 |
|------|------|
| **Marker 保持纯净** | PointMarker 只有 position/rotation，是纯空间标记 |
| **标注通过组件附加** | 符合 Unity 组件化设计，Inspector 天然支持 |
| **一点多用** | 同一 Marker 挂不同组件，被不同节点引用 |
| **Blueprint 保持纯净** | 只引用 Marker ID，不存储配置数据 |
| **导出时合并** | 空间数据 + 标注数据 → Playbook 条目 |
| **零额外 UI 开发** | 标注组件的 Inspector 由 Unity 自动生成 |

### 2.3 与现有模式的一致性

沿用项目中已有的**接口 + Attribute + AutoDiscover 注册表**模式：

| 已有系统 | 接口 | Attribute | 注册表 |
|---------|------|-----------|--------|
| Action 类型 | `IActionDefinitionProvider` | `[ActionType]` | `ActionRegistry` |
| Marker 类型 | `IMarkerDefinitionProvider` | `[MarkerDef]` | `MarkerDefinitionRegistry` |
| Marker 渲染 | `IMarkerGizmoRenderer` | — | `GizmoRenderPipeline` |
| **标注类型（新增）** | `IAnnotationDefinitionProvider` | `[AnnotationDef]` | `AnnotationDefinitionRegistry` |

## 二-A、两类 Annotation：手动配置型与工具生成型

> **v1.1 新增**（2026-02-26）

### 问题发现

在实际使用中发现，并非所有 Annotation 的数据都由策划手动填写。以 `SpawnAssignmentAnnotation` 为例：

- 数据来源：由 `PresetSpawnPointGeneratorTool` 从父级 `MonsterPoolAnnotation` 自动展开分配
- 策划操作：只需调整 SpawnPoint 的 Transform（位置/朝向），不应手动修改分配数据
- 视觉用途：在 SceneView 中显示每个点"被分配了什么类型的怪物"，辅助策划进行空间布局决策

这与 `CameraAnnotation`（策划手动填写 FOV、过渡时长等）的使用模式完全不同。

### 两类 Annotation 定义

| 维度 | 手动配置型 Annotation | 工具生成型 Annotation |
|------|----------------------|----------------------|
| **数据来源** | 策划在 Inspector 中手动填写 | 编辑器工具自动生成写入 |
| **Inspector 编辑性** | 可编辑 | 分配字段只读，行为字段可编辑 |
| **代表示例** | `CameraAnnotation`（FOV、缓动） | `SpawnAssignmentAnnotation`（MonsterType、Tag） |
| **语义** | "我对这个点的配置" | "这个点被分配了什么" |
| **修改方式** | 直接编辑字段 | 在父级重新运行生成工具 |
| **生命周期** | 策划手动创建/删除 | 工具批量创建，重新生成时全部覆盖 |

### 关键结论

两者在**数据模型层和导出管线中完全一致**——都是 `MarkerAnnotation` 子类，都通过 `CollectExportData` 收集导出数据。唯一的差异是 **Inspector 编辑权限**。

因此：
- **不需要新增框架层概念**（不需要新基类、不需要修改 sbdef 语法）
- **通过业务层 CustomEditor 解决**：分配字段只读，行为字段可编辑（方案 E）
- 未来如果工具生成型 Annotation 变多，可在 sbdef 语法中增加 `source tool` 标记让代码生成器自动输出只读 Editor

### Inspector 实现模式（方案 E：分字段只读 + 可编辑）

SpawnAssignment 的字段分两类：
- **分配字段**（MonsterType、Tag）：工具写入，`GUI.enabled = false` 只读
- **行为字段**（InitialBehavior）：工具写入默认值，策划可逐点调整

```csharp
[CustomEditor(typeof(SpawnAssignmentAnnotation))]
public class SpawnAssignmentAnnotationEditor : Editor
{
    public override void OnInspectorGUI()
    {
        serializedObject.Update();

        EditorGUILayout.HelpBox(
            "分配字段（类型/标签）由生成工具写入，初始行为可逐点调整。",
            MessageType.Info);

        // 分配字段：只读
        GUI.enabled = false;
        EditorGUILayout.PropertyField(serializedObject.FindProperty("MonsterType"));
        EditorGUILayout.PropertyField(serializedObject.FindProperty("Tag"));
        GUI.enabled = true;

        // 行为字段：可编辑
        EditorGUILayout.PropertyField(serializedObject.FindProperty("InitialBehavior"));

        serializedObject.ApplyModifiedProperties();
    }
}
```

### SpawnAssignmentAnnotation 字段修正（v1.1）

旧定义中 `MonsterId`（string）实际存储的是 `MonsterType`（int 1-8）——命名与语义不匹配。

| 旧字段 | 新字段 | 说明 |
|--------|--------|------|
| `string MonsterId` | `int MonsterType` | 怪物类型编号（1-8），运行时由 MonsterIdMapping 解析为真实 ID |
| `int Level` | `string Tag` | 分类标签字符串，从 MonsterPoolAnnotation 的枚举值 `.ToString()` 写入 |
| — | `string InitialBehavior` | 初始行为（Idle/Patrol/Guard/Ambush 等），工具写入默认值，策划可逐点调整 |

修正理由：
- **MonsterType 而非 MonsterId**：SceneBlueprint 中只有 MonsterType 概念，不同关卡中同一 Type 对应不同 MonsterId
- **Tag 而非 Level**：Level 不属于单点分配的职责；Tag 从 MonsterEntry 继承，辅助 Gizmo 可视化
- **int 而非 string**（MonsterType）：消除运行时不必要的 `int.TryParse` 转换
- **string 而非 enum**（Tag/InitialBehavior）：与 MonsterPool 的枚举类型解耦，避免生成重复枚举；为未来 Tag 系统演进留空间

### MonsterPoolAnnotation 字段调整（v1.1）

| 变更 | 说明 |
|------|------|
| 移除 `MinSpacing` | 最小间距不属于“怪物池”的职责，转移到 `PresetSpawnPointGeneratorTool`（工具自带 Slider 参数）和 `SpawnWaveSystem`（固定值，测试系统） |
| 新增 `HealthCoefficient` | MonsterEntryItem 新增 `float HealthCoefficient`（默认 1.0，最小 0.1），用于每种怪物类型的生命值缩放 |

### Tag 系统演进方向（备忘）

当前 `MonsterPool` 中的 `Tag` 是单维度枚举（`Normal/Elite/Boss/Minion/Special`），实际上混合了"品阶"和"角色定位"两个概念。未来可能演进为多维度正交标签系统：

```
维度1：品阶        → Normal / Elite / Boss
维度2：战斗角色    → Tank / DPS / Healer / Support  
维度3：攻击方式    → Melee / Ranged
维度4：职业(可选)  → Warrior / Mage / Archer / Shaman / ...

示例：萨满 = Ranged + Elite + Mage + Support
```

**当前设计的兼容性**：
- `SpawnAssignment.Tag` 使用 string 存储，无论未来 Tag 是单值、多值列表、还是 bitflag，该字段只需跟随序列化格式变化
- `MonsterPool.MonsterEntry.Tag` 当前为 enum，未来改为多维度标签时需要重构，但不影响 SpawnAssignment 侧
- 触发点：当业务上出现"按战斗角色过滤怪物池"或"给怪物同时标记多个属性"需求时，正式设计 Tag 系统

## 四、运行时层：MarkerAnnotation 基类

### 4.1 基类设计

```csharp
// Runtime/Markers/Annotations/MarkerAnnotation.cs

/// <summary>
/// 标记标注组件基类 — 附加在 SceneMarker 的 GameObject 上，
/// 为空间标记提供额外的语义属性。
///
/// 设计原则：
/// - Marker 是纯空间标记（position/rotation/shape）
/// - Annotation 是空间标注（monsterId/behavior/cameraFOV）
/// - 两者都属于 SceneView 层，不是 Blueprint 层
/// - 一个 Marker 可以挂多个不同类型的 Annotation
/// - 导出时，Annotation 数据和 Marker 空间数据合并写入 Playbook
/// </summary>
[RequireComponent(typeof(SceneMarker))]
public abstract class MarkerAnnotation : MonoBehaviour
{
    /// <summary>标注类型 ID（如 "Spawn", "Camera", "Patrol"）</summary>
    public abstract string AnnotationTypeId { get; }

    /// <summary>
    /// 收集导出数据 — 导出器调用此方法，将标注属性写入字典。
    /// key 是属性名，value 是可序列化的值。
    /// </summary>
    public abstract void CollectExportData(IDictionary<string, object> data);

    /// <summary>获取所在的 SceneMarker（缓存）</summary>
    public SceneMarker Marker => _marker ??= GetComponent<SceneMarker>();
    private SceneMarker _marker;

    // ── Gizmo 装饰（可选覆盖）──

    /// <summary>
    /// 是否提供自定义 Gizmo 装饰。
    /// 返回 true 时，管线会在 Marker 基础 Gizmo 之上调用 DrawGizmoDecoration。
    /// </summary>
    public virtual bool HasGizmoDecoration => false;

    /// <summary>
    /// 绘制 Gizmo 装饰。在 Marker 的基础 Gizmo（球体/箭头）之后调用。
    /// 仅在编辑器中有效（方法体由 Editor 层的 partial 或条件编译提供）。
    /// </summary>
    public virtual void DrawGizmoDecoration(bool isSelected) { }

    /// <summary>
    /// 覆盖 Marker 的 Gizmo 颜色。返回 null 表示不覆盖。
    /// </summary>
    public virtual Color? GetGizmoColorOverride() => null;
}
```

**关键设计决策**：

- `[RequireComponent(typeof(SceneMarker))]`：编译期保证标注组件必须挂在有 SceneMarker 的 GameObject 上
- `CollectExportData(IDictionary<string, object>)`：标注组件自己知道怎么导出，导出器不需要知道每种标注的具体字段
- Gizmo 方法是 `virtual`，默认空实现，具体标注按需覆盖
- 基类在 **Runtime 层**（因为 SceneMarker 在 Runtime 层，标注组件随场景序列化）

### 4.2 具体标注组件

#### SpawnAnnotation（刷怪点标注）

```csharp
// Runtime/Markers/Annotations/SpawnAnnotation.cs

/// <summary>
/// 刷怪点标注 — 标记该位置要生成什么怪物。
/// 挂在 PointMarker 上，配合 Spawn.Preset 节点使用。
/// </summary>
public class SpawnAnnotation : MarkerAnnotation
{
    public override string AnnotationTypeId => "Spawn";

    [Header("怪物配置")]
    [Tooltip("怪物模板 ID（对应运行时的怪物表）")]
    public string MonsterId = "";

    [Tooltip("怪物等级")]
    [Range(1, 100)]
    public int Level = 1;

    [Header("初始行为")]
    [Tooltip("生成后的初始行为")]
    public InitialBehavior Behavior = InitialBehavior.Idle;

    [Tooltip("警戒半径（Guard 模式下生效）")]
    public float GuardRadius = 5f;

    public override void CollectExportData(IDictionary<string, object> data)
    {
        data["monsterId"] = MonsterId;
        data["level"] = Level;
        data["behavior"] = Behavior.ToString();
        if (Behavior == InitialBehavior.Guard)
            data["guardRadius"] = GuardRadius;
    }

    // ── Gizmo ──

    public override bool HasGizmoDecoration => true;

    public override void DrawGizmoDecoration(bool isSelected)
    {
        // 在 PointMarker 上方画怪物 ID 标签
        if (!string.IsNullOrEmpty(MonsterId))
        {
            var pos = transform.position + Vector3.up * 1.2f;
            var label = $"{MonsterId} Lv.{Level}";
            // 使用 GizmoLabelUtil（编辑器层调用）
        }

        // Guard 模式画警戒范围
        if (Behavior == InitialBehavior.Guard)
        {
            // 画半透明圆盘
        }
    }

    public override Color? GetGizmoColorOverride()
    {
        return string.IsNullOrEmpty(MonsterId)
            ? null
            : new Color(0.9f, 0.2f, 0.2f); // 有怪物配置的点位用红色
    }
}

public enum InitialBehavior { Idle, Patrol, Guard, Ambush }
```

### 4.3 组件共存规则

| 规则 | 说明 |
|------|------|
| 一个 Marker 可挂多种不同类型的 Annotation | SpawnAnnotation + CameraAnnotation 可共存 |
| 同类型 Annotation 是否允许多个 | 由 `AnnotationDefinition.AllowMultiple` 控制 |
| 无 Annotation 的 Marker | 完全合法，是纯空间标记 |
| Annotation 不能脱离 Marker 存在 | `[RequireComponent]` 保证 |

## 五、编辑器层：标注定义注册表

### 5.1 AnnotationDefinition

```csharp
// Editor/Markers/Annotations/AnnotationDefinition.cs

/// <summary>
/// 标注类型定义 — 描述一种标注"是什么、适用于什么 Marker、有什么默认值"。
/// 用于驱动编辑器 UI（Add Component 菜单过滤、Inspector 提示等）。
/// </summary>
public class AnnotationDefinition
{
    /// <summary>标注类型 ID（如 "Spawn", "Camera"），对应 MarkerAnnotation.AnnotationTypeId</summary>
    public string TypeId;

    /// <summary>编辑器显示名（如 "刷怪点标注"）</summary>
    public string DisplayName;

    /// <summary>描述文本</summary>
    public string Description;

    /// <summary>对应的 MonoBehaviour 组件类型</summary>
    public Type ComponentType;

    /// <summary>
    /// 适用的 Marker 类型 ID 列表。
    /// null 或空数组表示适用所有 Marker 类型。
    /// </summary>
    public string[] ApplicableMarkerTypes;

    /// <summary>是否允许同一 Marker 上挂多个同类标注</summary>
    public bool AllowMultiple;

    /// <summary>关联的 Tag 前缀（用于图层系统集成）</summary>
    public string TagPrefix;
}
```

### 6.3 MarkerCache 缓存扩展

避免每帧 `GetComponents<MarkerAnnotation>()` 的 GC 开销：

```csharp
// MarkerCache 扩展
public static class MarkerCache
{
    // 现有
    private static SceneMarker[] _markers;

    // 新增：Annotation 缓存
    private static Dictionary<SceneMarker, MarkerAnnotation[]> _annotationCache;

    /// <summary>获取 Marker 上的所有 Annotation（从缓存读取，零 GC）</summary>
    public static MarkerAnnotation[] GetAnnotations(SceneMarker marker)
    {
        EnsureValid();
        if (_annotationCache != null && _annotationCache.TryGetValue(marker, out var annotations))
            return annotations;
        return Array.Empty<MarkerAnnotation>();
    }

    // 在 Rebuild() 中同时构建 Annotation 缓存
    private static void Rebuild()
    {
        _markers = FindObjectsOfType<SceneMarker>();
        _annotationCache = new Dictionary<SceneMarker, MarkerAnnotation[]>();
        foreach (var marker in _markers)
        {
            var annotations = marker.GetComponents<MarkerAnnotation>();
            if (annotations.Length > 0)
                _annotationCache[marker] = annotations;
        }
    }
}
```

缓存随 `EditorApplication.hierarchyChanged` 自动失效，与现有机制一致。

### 7.1 Spawn.Preset 的双绑定槽设计

`Spawn.Preset` 支持两种绑定源，策划按需选择：

```csharp
// SpawnPresetDef.cs 修改后
SceneRequirements = new[]
{
    // 方式 1：绑定区域（自动收集子 PointMarker）
    new MarkerRequirement("spawnArea", MarkerTypeIds.Area,
        displayName: "刷怪区域",
        defaultTag: "Combat.SpawnArea",
        required: false),

    // 方式 2：直接绑定点位
    new MarkerRequirement("spawnPoints", MarkerTypeIds.Point,
        displayName: "刷怪点位",
        allowMultiple: true,
        defaultTag: "Combat.SpawnPoint",
        required: false),
}
// 验证规则：spawnArea 和 spawnPoints 至少有一个非空
```

**策划的两种操作路径**：

```
路径 A（有区域，批量生成）：
  1. 创建 AreaMarker → 位置生成工具 → [随机生成] → 子 PointMarker
  2. Blueprint 中 Spawn.Preset 绑定 AreaMarker
  3. 导出时自动收集子 PointMarker + Annotation

路径 B（无区域，手动放置）：
  1. 手动放置若干独立 PointMarker
  2. Blueprint 中 Spawn.Preset 直接绑定这些 PointMarker
  3. 导出时直接使用

路径 C（混合）：
  1. 绑定一个 AreaMarker（批量点位）+ 几个独立 PointMarker（手动补充）
  2. 导出时合并两个来源
```

### 7.2 导出器收集逻辑

```csharp
/// <summary>
/// 统一收集 Spawn.Preset 节点关联的所有 PointMarker。
/// 支持 AreaMarker 子点位 + 直接绑定点位两种来源。
/// </summary>
List<PointMarker> ResolveSpawnPoints(ActionNodeData node)
{
    var result = new List<PointMarker>();

    // 来源 1：从 AreaMarker 收集子点位
    if (TryGetBinding("spawnArea", out AreaMarker areaMarker))
    {
        foreach (Transform child in areaMarker.transform)
        {
            var pm = child.GetComponent<PointMarker>();
            if (pm != null) result.Add(pm);
        }
    }

    // 来源 2：直接绑定的点位
    if (TryGetBindings("spawnPoints", out List<PointMarker> directPoints))
    {
        result.AddRange(directPoints);
    }

    return result;
}
```

导出每个 PointMarker 时收集 Annotation 数据：

```csharp
foreach (var pm in resolvedPoints)
{
    var entry = new Dictionary<string, object>
    {
        ["position"] = pm.transform.position,
        ["rotation"] = pm.transform.rotation.eulerAngles
    };

    // 收集所有 Annotation 数据
    var annotations = pm.GetComponents<MarkerAnnotation>();
    foreach (var annotation in annotations)
    {
        var annotationData = new Dictionary<string, object>();
        annotation.CollectExportData(annotationData);
        entry[annotation.AnnotationTypeId] = annotationData;
    }

    // 无 SpawnAnnotation 时打印日志
    if (pm.GetComponent<SpawnAnnotation>() == null)
    {
        Debug.Log($"[导出] PointMarker '{pm.MarkerName}' 没有 SpawnAnnotation，" +
                  $"将使用节点全局配置");
    }

    entries.Add(entry);
}
```

### 7.3 导出数据优先级

```
优先级（从高到低）：
  1. PointMarker 上的 SpawnAnnotation 属性    （逐点配置）
  2. Spawn.Preset 节点的全局属性              （全局配置）
  3. 默认值                                   （兜底）
```

**典型场景**：

| 场景 | 配置方式 |
|------|---------|
| 所有点位放同一种怪 | 不加 SpawnAnnotation，用节点全局怪物模板 |
| 不同点位放不同怪 | 每个 PointMarker 加 SpawnAnnotation |
| 大部分相同，个别特殊 | 全局配置 + 个别点位加 SpawnAnnotation 覆盖 |
